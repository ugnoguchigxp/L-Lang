# Static Judgment implementation plan

作成日: 2026-07-20

状態: **fixture実装・検証完了**

## 目的

コンパイル時に存在する具体的な自然言語値をConceptに照らして判断し、その結果を決定的なTypeScript定数へ固定する最小の縦切りを実装する。

```ts
const mike = staticValue(`
  三毛模様でニャーと鳴き、
  人間の家で飼育されている小型動物
`);

export const mikeIsCat = judgeStatic(mike, Cat);
```

生成物:

```ts
export const mikeIsCat = true as const;
```

同じConcept、値、Prompt versionに対しては、`semantic.lock`からAPIなしで同じ定数を再生成できなければならない。

## Predicate v2 Blockerとの関係

[`benchmarks/schema-evolution-v2/BLOCKER.md`](./benchmarks/schema-evolution-v2/BLOCKER.md)に記録した問題は、PredicateのConcept、Oracle、fixture境界に局在する。Static JudgmentはSchema Evolution v2、Predicate IRの`present`、型付きPredicate equivalence、2/3 Consensusを使用しないため、この実装を停止しない。

維持する制約:

- v2の`benchmark.json`と`freeze.json`を変更しない
- v2 live benchmarkを実行しない
- Static Judgment実装をv2の解決証拠として扱わない
- 共通の`semantic.lock`を拡張するとき、既存Predicate replayを壊さない

## 今回検証する仮説

[`concept.md`](./concept.md)の仮説1を対象とする。

> コンパイル時の意味的判断を、通常の定数として固定できる。

この段階では判断精度の一般化性能ではなく、次のコンパイラ経路が成立するかを検証する。

```text
Static literal + Concept
        ↓ build時のみresolver
true / false / unresolved
        ↓ validation / audit / lock
決定的なTypeScript定数
        ↓ replay
APIを使用しない同一生成物
```

## 固定するDSL

MVP sourceは、1つのConcept definition、1つの`staticValue`、1つの`judgeStatic`を持つ。

```ts
import { defineConcept, judgeStatic, staticValue } from "../../src/dsl";

const Cat = defineConcept("animal.cat")`
  A domesticated animal that is a biological cat.
`;

const mike = staticValue(`
  A small domesticated calico animal that meows.
`);

export const mikeIsCat = judgeStatic(mike, Cat);
```

`staticValue`が受理するのは、文字列literalまたはsubstitutionを含まないtemplate literalだけとする。

次はコンパイルエラーにする。

```ts
const description = await loadDescription();
const value = staticValue(description);
export const result = judgeStatic(value, Cat);
```

今回の段階検査は、一般的なデータフロー解析ではなく、次の構文制約で保証する。

- `staticValue`の引数がsource内のliteralである
- `judgeStatic`の第1引数が、そのsource内の`staticValue`宣言を参照する
- 第2引数が`defineConcept("stable.id")`宣言を参照する
- template substitution、関数呼び出し、property access、型assertionによる回避を拒否する

## resolver契約

Static Judgment専用のJSON Schemaを追加する。

```ts
type StaticJudgmentResolution =
  | {
      outcome: "resolved";
      value: boolean;
      diagnostics: string[];
    }
  | {
      outcome: "unresolved";
      value: null;
      diagnostics: string[];
    };
```

必須条件:

- `resolved`にはboolean値が必要
- `unresolved`には`null`が必要
- 入力に十分な情報がない場合は推測せず`unresolved`
- source dataをPrompt instructionとして扱わない
- networkや外部データへ依存した判断を要求しない
- raw response、parse結果、token usageをauditへ保存する

Prompt versionはPredicateと分離し、`STATIC_JUDGMENT_PROMPT_VERSION = "static-judgment-v1"`としてfingerprintへ含める。

## 生成物

判断名`mikeIsCat`に対して、sourceと同じdirectoryへ`mike-is-cat.generated.ts`を生成する。

```ts
// Generated by the semantic compiler. Do not edit.
export const mikeIsCat = true as const;
```

GeneratorはLLM textを使用せず、検証済みboolean値だけからコードを生成する。生成コードに次を含めてはならない。

- LLM clientまたはAPI key参照
- Concept text
- 元のStatic value
- resolver、Prompt、diagnostics
- runtime分岐

## lockfile設計

既存`semantic.lock` version 1のPredicate用`entries`を維持し、Static Judgment専用の`judgments` namespaceを追加する。既存コードが`entries`をPredicate entryとして扱う前提を崩さないため、単一mapのunionではなくnamespace分離を採用する。

```ts
type StaticJudgmentLockEntry = {
  fingerprint: string;
  source: string;
  judgment: string;
  conceptId: string;
  conceptHash: string;
  valueHash: string;
  promptHash: string;
  provider: string;
  model: string;
  resolvedValue: boolean;
  generatedCodeHash: string;
  response: ResponseMetadata | null;
  createdAt: string;
};
```

互換規則:

- 既存Predicate entryは従来どおり`entries`から読む
- Static Judgment entryは`judgments`から読む
- `judgments`が存在しない既存lockをそのまま読める
- Predicate replay検索は`judgments`を参照しない
- Judgment replay検索は`entries`を参照しない
- rootの`semantic.lock`を計画実装中に一括migrationしない
- version bumpは既存形式を読めなくする変更が必要になった場合だけ行う

Judgment replay key:

- source path
- judgment name
- Concept IDとConcept hash
- Static value hash
- Prompt hash

providerとmodelは、既存Predicate replayと同様にreplay検索条件へ含めない。

## auditとtransaction

auditは`.semantic/judgments/<run-id>/`へ保存する。

| File | 内容 |
| --- | --- |
| `input.json` | source、Judgment名、Concept ID/hash、value hash、Prompt hash、provider/model |
| `response.output.json` | fixtureまたはAPIのraw output |
| `judgment.json` | 検証済みの`true`、`false`またはunresolved diagnostics |
| `candidate.ts` | 決定的に生成した定数候補 |
| `report.json` | stage、cache/replay、hash、duration、失敗情報 |

transaction順序:

1. source scanと段階検査
2. fingerprint計算とlock lookup
3. lock missの場合だけresolver実行
4. resolver outputの厳格parse
5. deterministic code生成
6. candidate typecheck
7. project typecheck
8. final生成物を原子的にpromotion
9. full test
10. 成功後だけlock更新
11. report保存

失敗時:

- unresolvedではfinal生成物とlockを変更しない
- typecheckまたはfull test失敗では以前のfinal生成物へrollbackする
- lock書き込み失敗ではpromotionをrollbackする
- 一時candidateは`finally`で削除する

## CLI

既存の利用形を維持する。

```bash
bun run semantic build examples/static-judgment/semantic.ts \
  --fixture examples/static-judgment/openai-response.true.fixture.json

bun run semantic replay examples/static-judgment/semantic.ts
```

`semantic build`と`semantic replay`はsource formをsyntax-onlyで判定し、既存Predicate compilerまたはStatic Judgment compilerへdispatchする。

今回、`test`、`check`、`diff`、`approve`はStatic Judgment sourceを受理しない。対象外であることが分かるerrorを返す。
